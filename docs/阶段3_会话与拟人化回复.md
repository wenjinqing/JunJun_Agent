# 阶段 3 · 会话与拟人化回复

> 周期：2-3 天
> 目标：消息入库、persona 全量、回复后处理（分条/错别字/引用/延迟）、频率控制全量——回复「观感」与原 bot 一致。
> 修订（2026-07-16）：由原「阶段3 功能复刻主体」拆出。回复后处理与 VLM 识图为原计划遗漏项。

## 一、技术要点

### 1.1 数据库落地 `junjun_core/database`
- peewee models 实装并建表（对齐原 `database_model.py`，全部带 `bot_id` 字段，默认当前 QQ 号）：
  - `Messages`（含 `is_mentioned/is_at/reply_to/processed_plain_text/key_words/interest_value`）
  - `ChatStreams`、`LLMUsage`、`Images`、`PersonInfo`/`Jargon`/`Expression`/`Emoji`/`ReminderTasks`（声明即可，阶段 4/5 用）。
- SQLite WAL 模式 + 单写协程队列（原项目踩过并发写锁）。
- 消息入库：入站消息与 bot 回复都落 `Messages`（对齐原 `MessageStorage.store_message`，剥离 `<MainRule>` 类标签逻辑不再需要——新 prompt 不混入标签）。
- `LLMUsage` 落库：从阶段 2 的日志提取改为写表（model/prompt_tokens/completion_tokens/cost/request_type）。

### 1.2 多模态入站（原计划遗漏）
- VLM 识图链路（对齐原 `utils_image.py`）：图片段 → base64 → 任务槽 `vlm` 描述 → `Images` 表（hash 去重）→ 上下文渲染为 `[图片：描述]`。
- 表情包入站识别：`is_emoji` 标记 + 简述进上下文。
- ASR（原配置 `enable_asr=false`）：接口预留（`voice` 任务槽占位），不实装。
- 识图失败降级：`[图片]` 占位，不阻塞回复。

### 1.3 Persona 全量 `junjun_agent/persona`
- 迁移原 `[personality]`：personality / reply_style / interest / plan_style（`v2_append_legacy_plan_style_to_system_prompt` 语义并入构造器）/ visual_style。
- prompt 构造器分块拼装：人设 + reply_style + 当前时间 + 群/私聊场景块 + 情绪占位（阶段 5 接 mood）+ 记忆摘要占位（阶段 4 接）+ 关系占位。
- 保留 `_strip_emoji_for_system_prompt`（emoji 干扰 function calling schema，原项目实测坑）。
- keyword_reaction 迁移：命中关键词规则 → 规则提示注入 prompt（对齐原 replyer 内联语义）。

### 1.4 回复后处理 `junjun_agent/postprocess`（原计划遗漏，拟人化核心）
流水线：agent 原始文本 → 后处理 → `ReplySet(segments=[...])`：
1. **response_splitter**（对齐原 `[response_splitter]`）：按句拆多条，`max_sentence_num` 超限时 `enable_overflow_return_all` 一次全发；`enable_kaomoji_protection` 颜文字保护开关。
2. **chinese_typo**（对齐原 `[chinese_typo]` 全参数）：拼音同音字替换错别字生成器，概率参数迁移原配置。
3. **引用回复决策**：`reply_message_quote="llm"` 语义——agent 输出结构含 `quote_message_id`（prompt 约定或 tool 参数），后处理落 `reply_to_message_id`。
4. **分条延迟发送**：多 segment 按字数模拟打字延迟逐条发（send_handler 侧 sleep），间隔加抖动。
- 全流水线纯函数化，单测覆盖拆分/错别字概率/颜文字保护。

### 1.5 频率控制全量 `junjun_agent/funnel`（升级阶段 2 的 L1）
- `talk_value_rules` 时段规则（`enable_talk_value_rules`）：按时段乘系数。
- **LLM 动态频率调节**（对齐原 `FrequencyControl.trigger_frequency_adjust`）：≥160s 且 ≥20 条消息时，`utils` 小模型评「过于频繁/过少/正常」→ `talk_frequency_adjust *= 0.8/1.2`，夹在 `[0.1, 1.5]`。
- `frequency_boost_when_addressed`：昵称直呼加成；`reply_probability_boost` 消息级加成透传。
- Timing Gate（`enable_timing_gate`，默认关）：开启时新消息先等 `timing_gate_wait_seconds` 聚拢，窗口内只评估一次，超时强制 continue。
- 连续不回复计数 + `decision_reset_noreply_on_action`。
- 会话级串行队列：同会话消息按序处理，LLM 处理中堆积的消息合并进上下文、只触发一次决策；超 60s 过期丢弃打 WARN。

## 二、避坑与潜在 Bug

| 风险 | 说明 | 对策 |
|------|------|------|
| SQLite 并发写锁 | 多协程同时写库 `database is locked` | WAL + 单写队列；写操作全走 executor |
| 错别字生成毁专有名词 | URL/数字/代码被替换 | 白名单跳过（URL/数字/英文串）；概率参数对齐原配置 |
| 分条延迟阻塞会话 | sleep 在主循环卡其他会话 | 延迟发送在 send_handler 侧异步任务执行 |
| 引用消息 ID 失效 | quote 的 message_id 已撤回/过期 | 发送失败降级为不带引用重发 |
| VLM 识图慢 | 图片描述 10s+，消息堆积 | 识图带 15s 超时，超时降级 `[图片]`；结果缓存（hash 去重）|
| 频率调节双向震荡 | LLM 忽高忽低来回调 | 调整夹 `[0.1,1.5]` + 160s 冷却（对齐原实现）|
| timing gate 死锁 | wait 态等新消息，新消息又触发 wait | 超时强制 continue；窗口内只评一次 |
| 合并消息炸上下文 | forward 段递归展开超长 | 递归深度限 2，展开文本截断 500 字 |
| prompt 场景块串台 | 群聊块用到私聊 | persona 构造器按 session.is_group 分支，单测覆盖两分支 |
