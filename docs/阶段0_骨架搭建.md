# 阶段 0 · 骨架搭建

> 周期：1 天
> 目标：建立 monorepo workspace、空包骨架、统一入口，跑通空网关启动。

---

## 一、技术要点

### 1.1 monorepo workspace 骨架
- 根 `pyproject.toml` 用 `[tool.uv.workspace]` 或 `[tool.hatch]` 聚合子包。
- 每个子包 `junjun_*` 各自有 `pyproject.toml`，声明 `dependencies` 仅依赖**下层**包。
- Python 版本锁定 `>=3.10`（与原项目一致）。
- 用 `uv` 管理依赖（原 MaiBot-Napcat-Adapter 已用 `uv.lock`）。

### 1.2 配置体系骨架
- `config/bot_config.toml`：从原 `MaiBot/config/bot_config.toml` 抽取结构，**字段名对齐**（`talk_value`/`mentioned_bot_reply`/`max_context_size` 等），便于后续迁移逻辑直接复用。
- `.env.example`：列出所有环境变量（`MAIBOT_QQ_ACCOUNT`、各 API key、`LANGFUSE_*`）。
- `junjun_core/config/`：统一配置加载入口 `load_config()`，支持 toml + env 变量插值（`${VAR}` 语法，对齐原项目）。

### 1.3 统一入口
- `scripts/run_junjun.py`：负责 `chdir` 到仓库根、`load_dotenv`、初始化日志、启动网关。
- 对齐原 `launch/junjun.py` 的 `_setup_env()` 行为（工作目录、env、日志初始化）。
- 优雅关闭：注册信号处理，取消所有 asyncio task（对齐原 `graceful_shutdown`）。

### 1.4 可观测骨架
- `junjun_core/observability/langfuse_client.py`：迁移原 `lf` 单例，未配置时 `_NoopSpan` 空操作。
- `junjun_core/observability/logger.py`：structlog + rich，对齐原 `common/logger.py` 接口 `get_logger(name)`。

### 1.5 数据库骨架
- `junjun_core/database/`：peewee models 占位（`Messages`/`ChatHistory`/`ReminderTasks`/`LLMUsage`），表结构对齐原 `database_model.py`。
- 暂不建表，仅声明，阶段 3 再迁移逻辑。

## 二、避坑与潜在 Bug

| 风险 | 说明 | 对策 |
|------|------|------|
| 工作目录错位 | 原项目依赖 `os.chdir(_ROOT)`，相对路径读取配置/模板 | 入口固定 `chdir` 到仓库根；所有路径用 `Path(__file__).resolve().parent` 推导 |
| env 变量插值失败 | 原 `bot_config.toml` 用 `${MAIBOT_QQ_ACCOUNT}` 语法 | 配置加载器必须实现 `${VAR}` 替换，缺失时给明确报错而非空字符串 |
| 循环 import | `observability` 被多处引用，若它反向 import 业务包会循环 | observability 只依赖标准库 + langfuse SDK，零业务依赖 |
| workspace 子包互相找不到 | monorepo 首次配置不当导致 `import junjun_core` 失败 | 用 editable install（`uv pip install -e .`）或 workspace path 自动注入 |
| 日志中文乱码 | Windows 控制台默认 GBK | `sys.stdout.reconfigure(encoding='utf-8')` 或 rich 自动处理 |
| venv 混用 | 原项目有 `.venv` 和 `venv` 两套 | 新仓库只建一个 `.venv`，写进 `.gitignore` |
