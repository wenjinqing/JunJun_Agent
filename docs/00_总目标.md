## 一、升级目标

将君君 QQ 聊天机器人由原 MaiBot（Astron bot 架构）升级为**现代 AGENT 架构**，完整复刻原项目全部功能，技术栈如下：

| 维度 | 原项目 | 升级后 |
|------|--------|--------|
| 决策架构 | 手写 planner/replyer + HeartF/Brain 双轨 + chat_v2 半成品 | **三级决策漏斗**：规则门 → 小模型 Gate → LangChain 1.x `create_agent`（LangGraph runtime）|
| 工具调用 | 自研 `BaseTool` + `ToolExecutor` | `@tool` Function Calling + Skill 注册表 |
| 外部能力 | 单个 relationship MCP server | MCP 客户端（`langchain-mcp-adapters` 多 server）+ 保留 relationship 服务端 |
| LLM 抽象 | 自研 `llm_models/utils_model.py` | `langchain_openai.ChatOpenAI` 分任务模型（`init_chat_model`）|
| 可观测 | `langfuse_client.py` 空操作降级 | Langfuse **v3 SDK** `CallbackHandler` + `@observe`（对齐自托管 v3 服务器）|
| 接入层 | `maim_message` WS | 复用 `maim_message`，仅迁协议层 |

> **技术选型硬约束**：LangChain 取 **1.x 稳定线**（`create_agent`，弃用 `AgentExecutor`/`create_openai_tools_agent` 旧 API）；Langfuse SDK 取 **v3**（`E:\MaiM\langfuse` 自托管为 v3 镜像）。理由：新项目从弃用 API 起步 = 落地即遗留代码。

## 二、必须复刻的原项目功能清单

勘探 `E:\MaiM\MaiBot-JunJun\MaiBot\src` 确认以下功能必须 1:1 复刻：

1. **QQ 接入**：WS 协议（NapCat 8095 → Bot 8092）、黑白名单（群/私聊/ban_user）、戳一戳、@ 解析、引用回复、合并转发解析。
2. **聊天决策**：三级漏斗（名单/频控规则 → Gate 小模型 reply/no_reply → 主 Agent）；Timing Gate、talk_value 频率控制（含 LLM 动态调节）、沉默模式（no_reply_until_call）、连续不回复计数、`mentioned_bot_reply` @ 必回旁路。
3. **会话管理**：群聊 HeartF / 私聊 Brain 语义统一为 `ChatSession`（差异靠 prompt + skill 集合，不分支代码）。
4. **回复后处理**（原 `[response_post_process]`，拟人化核心，易漏）：response_splitter 长回复拆多条 + 分条延迟发送、chinese_typo 错别字生成器、引用回复决策（`reply_message_quote="llm"`）。
5. **工具系统**：时间/天气/搜索/记忆查询/提醒/发送消息/表情等内置工具 + TTS + repeat 主动复读动作 + vrchat_agent 插件。
6. **记忆系统**：短期窗口 + 长期检索（faiss bge-m3 1024 维）+ hippo_memorizer 话题摘要压缩 + 用户画像（memory_points）+ 知识图谱（LPMM：KG/openIE/quick_algo）+ MemoryForgetTask 遗忘。
7. **多模态入站**：VLM 识图（图片→描述→`[picid]` 入库→上下文替换）、表情包语义识别；ASR 语音识别（原配置关闭，接口预留）。
8. **主动系统**：主动私聊话题（proactive_topics 去重）、repeat 复读参与、提醒任务（ReminderTaskManager 持久化 + 重启恢复）、定时统计（OnlineTimeRecord/StatisticOutput）。
9. **情绪/表情包/黑话/表达学习**：mood 情绪状态进 prompt、emoji 库（steal_emoji 偷图注册 + 定期扫描 + 用量统计）、jargon 黑话识别与记忆、expression learner+**selector**+reflector 三件套（含 expression_groups 跨会话共享）。
10. **关系系统**：relationship（`build_relationship` 进 prompt）+ keyword_reaction 关键词规则。
11. **MCP**：relationship server（保留为服务端，工具：apply_relationship_penalty / update_user_impression / add_user_tag / set_user_name）。
12. **WebUI**：配置/日志/统计/插件管理（FastAPI）。
13. **可观测**：Langfuse trace + 结构化日志 + token 用量统计（LLMUsage 落库）+ DB 自动清理。

> **明确不复刻**：HeartF/Brain 双轨与 chat_v2 半成品（被单一 Agent 取代）、多 bot 实例机制（BotInstance/BotScopedDB，见约束 8）、telemetry 上报、pixiv_novel 等边缘数据、旧 8000 端口 legacy server。

## 三、目标架构总览

```
┌─────────────────────────────────────────────────────────┐
│  接入层 Adapter                                          │
│  junjun_adapter_napcat  (maim_message WS)               │
│  ← NapCat.Shell (ws://127.0.0.1:8095)                   │
└───────────────┬─────────────────────────────────────────┘
                │ UnifiedMessage
┌───────────────▼─────────────────────────────────────────┐
│  消息网关 Gateway                                        │
│  junjun_core/gateway  路由/会话/黑白名单/速率            │
└───────────────┬─────────────────────────────────────────┘
                │ ChatSession
┌───────────────▼─────────────────────────────────────────┐
│  决策漏斗（省 token 的关键，逐级拦截）                    │
│  L1 规则门: 名单/自消息/talk_value 概率/@旁路  (0 token) │
│  L2 Gate:   小模型 reply|no_reply|until_call  (小 token) │
│  L3 Agent:  junjun_agent  LangChain create_agent         │
│  ├ persona(人设/prompt)                                  │
│  ├ memory(短期/长期/检索)                                │
│  ├ tools(skill 注册表 + MCP 工具)                         │
│  ├ postprocess(分条/错别字/引用/延迟)                     │
│  └ loop(定时任务→主动/提醒/统计/遗忘)                     │
└──┬───────────┬───────────────┬───────────────┬─────────┘
   │           │               │               │
   ▼           ▼               ▼               ▼
 skill层    memory层         LLM层          MCP/外部
junjun_     junjun_          junjun_         junjun_
skills      memory           llm             mcp_client
                             ↑
                      Langfuse 观测层（横切，位于 junjun_core/observability）
```

### 依赖原则（拒绝高耦合）
- 箭头**单向向下**，禁止逆向 import。
- Adapter 不知 Agent 存在；Agent 不知具体 LLM 厂商；Skill 不直接 import Agent 内部。
- 层间只通过**数据契约**通信：`UnifiedMessage`、`ChatSession`、`ExecutionResult`、`Skill` 接口。
- 每个子包是独立 Python 包，通过 workspace 互相依赖，**禁止跨层文件引用**。

## 六、关键约束（拒绝屎山）

1. **不复制原项目源码目录**：仅迁移逻辑，包名统一 `junjun_*`，消除 `src/`+`chat/`+`chat_v2/` 三套并存历史包袱。
2. **单一 Agent 架构**：不再保留 HeartF/Brain 双轨，原 `chat_v2` 半成品废弃；决策差异全部收敛到「三级漏斗 + persona + skill 集合」。
3. **LangChain 1.x 作为编排骨架**：`create_agent`（LangGraph runtime），禁止使用已弃用的 `AgentExecutor`/`create_openai_tools_agent`；决策语义（reply/no_reply/动作）通过漏斗 + skill 表达。
4. **Langfuse v3 横切**：观测代码集中在 `observability` + `junjun_llm/tracing`，业务层不直接持有 Langfuse 句柄；SDK 与自托管 v3 服务器版本对齐。
5. **MCP 双向**：君君既是 MCP 客户端（调外部），也保留 relationship server 作服务端。
6. **NapCat 不动**：`E:\MaiM\NapCat.Shell` 配置保持，Adapter 指向新网关端口 8092。
7. **可降级**：Langfuse/SiliconFlow/Google 任一不可用时静默降级，不阻塞主循环。
8. **单 bot + 预留隔离字段**（2026-07-16 决策）：不迁移 BotInstance/BotScopedDB/MessageRouter 多实例机制；但所有 DB 表和会话键保留 `bot_id` 字段（默认取当前 QQ 号），未来多 bot 低成本扩展。
9. **存量数据不迁移**（2026-07-16 决策）：新君君从零积累记忆/画像/表情包，不写旧库迁移脚本；旧 `MaiBot.db` 与 `data/` 仅作只读参考。
10. **旧 bot 已停机**：无双回复风险，新 Adapter 直连 NapCat 8095 开发调试。
11. **每阶段带测试**：pytest + pytest-asyncio，漏斗/契约/后处理等纯逻辑必须有单测；`uv run pytest` 全绿才算阶段完成。
