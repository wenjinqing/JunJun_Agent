## 一、技术要点

### 1.1 三层记忆模型 `junjun_memory`
| 层 | 来源 | 实现 |
|----|------|------|
| 短期 | 阶段 2 滑动窗口 | 已有，升级为 LangGraph checkpointer（会话内多轮工具调用状态持久）|
| 中期（话题摘要）| 原 `hippo_memorizer/chat_history_summarizer.py`（heartFC 实际在用）| `summarizer.py`：消息批次 → LLM 话题提取 → 话题缓存累积 → 压缩摘要落库；数据 `data/hippo/<chat_id>.json` 语义对齐 |
| 长期（向量检索）| 原 `memory_system/memory_retrieval.py` | faiss-cpu + bge-m3（SiliconFlow，1024 维），索引落盘 `data/embedding/`，启动 load |

- 记忆检索暴露两种形态：LangChain Retriever（prompt 注入近期相关记忆摘要）+ skill（`recall_memory` 主动查）。
- MemoryForgetTask（对齐原 `chat/utils/memory_forget_task.py`）：定时衰减/清理低权重记忆。

### 1.2 LPMM 知识库 `junjun_memory/knowledge`
- 迁移原 `chat/knowledge`：`embedding_store.py`（faiss+JSON 元数据）、`kg_manager.py`（quick_algo DiGraph + PageRank，`data/rag/rag-graph.graphml`）、`open_ie.py`+`ie_process.py`（三元组抽取）、`mem_active_manager.py`。
- 配置对齐 `[lpmm_knowledge]`：`embedding_dimension=1024`、`rag_synonym_search_top_k=10`、`qa_ppr_damping=0.8` 等。
- 任务槽补齐：`embedding`、`lpmm_entity_extract`、`lpmm_rdf_build`、`lpmm_qa`。
- **quick_algo 是私有依赖**：先验证新 venv 可装可用；装不上则 PageRank 降级 networkx 实现（接口不变）。
- skill：`search_knowledge`。

### 1.3 用户画像 `junjun_memory/user_profile.py`
- 对齐原 `person_info/Person`：`person_id=MD5(platform+user_id)`、`person_name`、`memory_points`（`分类:内容:权重` JSON）。
- `build_relationship()` 进 persona prompt（阶段 3 留的关系占位在此接上）。
- 写路径：skill `manage_user_profile`（LLM 实时写）+ 阶段 6 的 relationship MCP 工具复用同一存储。
- 并发写：peewee 事务 + memory_points 字段级 merge（不覆盖全量）。

### 1.4 黑话 `junjun_express/jargon.py`
- 对齐原 `jargon/jargon_explainer.py`：`match_jargon_from_text`（上下文渲染时黑话标注解释）+ 检索时黑话概念展开。
- `enable_jargon_detection`、`all_global` 配置对齐；Jargon 表计数累积。
- skill：`query_jargon`（供 agent 查黑话含义）。

### 1.5 记忆类 skill 全量 `junjun_skills/builtin`
- `recall_memory` / `save_memory` / `search_knowledge` / `query_jargon` / `manage_user_profile` / `query_chat_history`（对齐原 memory_retrieval 的 8 工具中仍有价值者；原 ReAct 检索 agent 不再单独实现——主 agent 直接持有这些工具，`max_agent_iterations` 语义由 recursion_limit 承接）。

## 二、避坑与潜在 Bug

| 风险 | 说明 | 对策 |
|------|------|------|
| quick_algo 装不上 | 私有图算法库，新环境可能缺 wheel | 先探测；降级 networkx PageRank，接口隔离在 kg_manager 内 |
| faiss 索引与元数据错位 | 索引重建后 id 映射错 | 索引+metadata 原子成对落盘；启动校验数量一致，不一致重建 |
| embedding 维度错配 | 换模型后维度不同炸索引 | 索引文件头记录维度+模型名，不匹配拒绝加载并提示重建 |
| 摘要任务烧 token | 每条消息都触发摘要 | 批次触发（对齐原 hippo：按条数+时间窗）；`utils` 小模型 |
| checkpointer 膨胀 | LangGraph 状态无限增长 | 窗口裁剪仍生效；checkpointer 只存活跃会话，LRU 淘汰 |
| 画像并发写丢失 | 多会话同时更新同一用户 | 事务 + 字段级 merge |
| 黑话误判 | 正常词被永久标记 | 阈值宽松 + WebUI 可删（阶段 7）|
| 记忆检索拖慢回复 | 同步检索阻塞 agent | Retriever 注入限 top-k=5 + 200ms 超时降级空 |
| 旧数据混入 | 误读旧 MaiBot.db | 从零积累（已决策）；代码不含旧库路径 |

## 三、验收标准

- [ ] pytest 新增：摘要批次触发/faiss 存取/画像 merge/黑话匹配 单测全绿。
- [ ] 聊 20+ 条后自动产出话题摘要（日志+落盘可见）。
- [ ] "你还记得 XX 吗" → `recall_memory` 命中此前聊过的内容。
- [ ] 知识库导入一篇文档后，相关提问 `search_knowledge` 能答。
- [ ] 君君能记住"我叫XX/我喜欢XX"并在画像中体现，下次对话 prompt 带关系信息。
- [ ] 黑话开启后，群内高频黑话被记录并在上下文中标注解释。
- [ ] 重启后 faiss 索引、画像、摘要全部恢复。
